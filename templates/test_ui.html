Who Uses RAG in Production?

RAG is the dominant architecture for production AI systems that need grounded, up-to-date answers.

Google
Search Generative Experience uses RAG to ground AI Overviews in real-time web content rather than stale training data.
GitHub Copilot
Retrieves from your open files, imports, and repository context before generating code suggestions.
Perplexity AI
Built entirely on RAG. Every answer cites sources. Valued at $9B+ — built on retrieval, not fine-tuning.
Amazon
Customer service bots use RAG to retrieve from product manuals, handling millions of queries with cited answers.
Microsoft 365 Copilot
Retrieves from emails, documents, and Teams chats via Microsoft Graph before generating responses.
Notion AI
Searches your workspace pages and databases as context before answering — your knowledge base as a RAG corpus.

Stage-by-Stage Deep Dive

Stage 1 — Document Loading
src/load_docs.py

Read .txt files from data/ into a dict[str, str] mapping filenames to content.

def load_documents(folder_path: str) -> dict[str, str]
def validate_folder(folder_path: str) -> bool
Common pitfall: Using open() without encoding="utf-8" — fails on Windows with non-ASCII. Use Path.read_text(encoding="utf-8").
Stage 2 — Chunking
src/chunking.py

Sliding window with step = chunk_size - overlap. Overlap ensures sentences at boundaries appear in both adjacent chunks.

def chunk_text(text: str, chunk_size: int = 300, overlap: int = 50) -> list[str]
def chunk_all_documents(documents: dict, chunk_size: int = 300, overlap: int = 50) -> list[dict]
Common pitfall: Off-by-one errors in the sliding window. The last chunk may be shorter than chunk_size — that's expected.
Stage 3 — Embeddings
src/embeddings.py

model.encode() returns numpy arrays. Call .tolist() to convert to Python lists for ChromaDB and JSON.

def load_embedding_model(model_name: str = "all-MiniLM-L6-v2") -> Any
def embed_texts(model, texts: list[str]) -> list[list[float]]
def embed_query(model, query: str) -> list[float]
Common pitfall: Returning numpy arrays instead of Python lists. ChromaDB expects list[list[float]].
Stage 4 — Vector Storage
src/vector_store.py

ChromaDB stores vectors + metadata together. Use get_or_create_collection for idempotency.

def get_chroma_client(persist_directory: str = "chroma_db") -> Any
def create_collection(client, collection_name: str = "dholakpur_helpdesk") -> Any
def add_chunks(collection, chunks: list[dict], embeddings: list[list[float]]) -> None
Common pitfall: Using create_collection instead of get_or_create_collection — fails on second run.
Stage 5 — Retrieval
src/retrieve.py

collection.query() returns nested lists (batch API). Flatten into [{text, source, chunk_id, distance}].

def retrieve_chunks(collection, query_embedding: list[float], n_results: int = 5) -> list[dict]
def format_context(chunks: list[dict]) -> str
Common pitfall: ChromaDB returns {"documents": [[...]]} — double nesting. Access with [0] for single queries.
Stage 6 — Generation
src/generate.py

The system prompt constrains the LLM to only answer from retrieved context — the primary anti-hallucination measure.

def build_prompt(query: str, context: str) -> str
def call_llm(prompt: str, api_key: str, model: str = "gpt-3.5-turbo", base_url: str | None = None) -> str
def generate_answer(query: str, context: str, api_key: str, ...) -> str
Common pitfall: Not including an explicit grounding instruction. Without it, the LLM will invent policies.

Chunking Strategies Compared

StrategyHow it worksProsCons
Fixed-size (this workshop) Sliding window of N characters with overlap Simple, predictable, no dependencies Splits mid-sentence, no semantic awareness
Sentence-basedSplit on sentence boundariesPreserves sentence integrityUneven sizes, needs NLP library
Recursive (LangChain)Try separators: \n\n, \n, sentence, charAdapts to document structureComplex, framework-dependent
Semantic chunkingEmbed sentences, group by similarityTopic-coherent chunksExpensive (embeddings for chunking itself)
