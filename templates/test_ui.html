How the RAG Pipeline Works

Each file in src/ handles one stage. You implement them in order — each stage depends on the one before it.

Stage 1 — Document Loading

File: src/load_docs.py

Read the three .txt files from data/ into a Python dictionary. Keys are filenames, values are file contents.

from src.load_docs import load_documents
docs = load_documents("data")
# → {"attendance_policy.txt": "...", "college_handbook.txt": "...", ...}

Key concepts: Use pathlib.Path for file operations. The .glob("*.txt") method finds files matching a pattern. Read with .read_text(encoding="utf-8").

Stage 2 — Chunking

File: src/chunking.py

Split each document into smaller overlapping windows. This is needed because embedding models work best on focused, short text — not entire documents.

from src.chunking import chunk_text
chunks = chunk_text("A long document...", chunk_size=300, overlap=50)
# → ["chunk 1 (300 chars)", "chunk 2 (overlaps by 50 chars)", ...]

Key concepts: A sliding window with step size = chunk_size - overlap. The overlap ensures sentences at boundaries appear in both adjacent chunks.

Stage 3 — Embeddings

File: src/embeddings.py

Convert text into 384-dimensional vectors using all-MiniLM-L6-v2. Similar meanings produce nearby vectors — this is what enables semantic search.

from src.embeddings import load_embedding_model, embed_texts
model = load_embedding_model()
vectors = embed_texts(model, ["attendance policy", "canteen hours"])
# → [[0.023, -0.15, ...], [0.089, 0.04, ...]]  (384 floats each)

Key concepts: model.encode(texts) returns numpy arrays. Call .tolist() to convert to Python lists. Queries and documents must use the same model.

Stage 4 — Vector Storage

File: src/vector_store.py

Store chunk text, vectors, and metadata in ChromaDB. Persistence means you only embed once — future runs load from disk.

from src.vector_store import get_chroma_client, create_collection, add_chunks
client = get_chroma_client()  # Persists to chroma_db/
collection = create_collection(client)
add_chunks(collection, chunks, embeddings)

Key concepts: collection.add(ids=..., documents=..., embeddings=..., metadatas=...). Each chunk needs a unique ID and metadata containing its source filename.

Stage 5 — Retrieval

File: src/retrieve.py

Given a question, embed it and query ChromaDB for the most similar chunks. Format the results into a context string with source citations.

from src.retrieve import retrieve_chunks, format_context
results = retrieve_chunks(collection, query_embedding, n_results=5)
context = format_context(results)
# → "[Source: attendance_policy.txt]\nMinimum attendance is 75%...\n---"

Key concepts: collection.query(query_embeddings=[vec], n_results=5) returns nested lists. Results include documents, metadatas, and distances.

Stage 6 — Generation

File: src/generate.py

Construct a prompt that includes a system instruction (answer only from context, cite sources), the retrieved context, and the user's question. Call an OpenAI-compatible API.

from src.generate import build_prompt, call_llm
prompt = build_prompt(query="What is the min attendance?", context=context)
answer = call_llm(prompt, api_key="...", model="gpt-3.5-turbo")

Key concepts: The system instruction prevents hallucination. The base_url parameter lets you point at Ollama or any compatible server. Messages use [{"role": "user", "content": prompt}].

Running the Full App

File: src/app.py

Wire all stages together. Two modes:

# Index documents (run once)
python -m src.app --build

# Ask questions interactively
python -m src.app

Setup: Copy .env.example to .env and set your OPENAI_API_KEY. For Ollama, set OPENAI_BASE_URL=http://localhost:11434/v1 and OPENAI_MODEL=llama3.
